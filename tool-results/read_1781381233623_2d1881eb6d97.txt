   530→health endpoint. Alert channels include email and Slack integration.
   531→This provides basic availability monitoring sufficient for the launch
   532→phase, with migration to Vercel\'s built-in analytics and Synthetics on
   533→the Pro plan as the platform scales.

   535→[]{#_Toc100011 .anchor}**2.6 Data Resilience**
   536→
   537→Data resilience in RxBD operates at three levels, ensuring that no
   538→single point of failure can result in permanent data loss.
   539→
   540→[]{#_Toc100012 .anchor}**2.6.1 Local-First Auto-Save**
   541→
   542→Prescription drafts are automatically persisted to the browser\'s
   543→localStorage every 10 seconds using a debounced Zustand middleware. The
   544→auto-save captures the full prescription state including patient
   545→context, medications added, clinical notes, and form dirty state. Upon
   546→reconnection after a connectivity loss, the application detects local
   547→drafts by comparing the localStorage timestamp against the server\'s
   548→last-known save time and prompts the user to resume their work. This
   549→mechanism prevents data loss even during extended internet outages,
   550→which are common in semi-urban Bangladesh, and requires zero server
   551→infrastructure.
   552→
   553→[]{#_Toc100013 .anchor}**2.6.2 Database Backup Strategy**
   554→
   555→The primary backup mechanism uses Supabase\'s built-in daily automated
   556→backups, which are available on the Pro plan. During the free-tier
   557→launch phase, a GitHub Actions workflow performs a daily pg_dump of the
   558→Supabase database, encrypts the dump using AES-256 encryption with a key
   559→stored in GitHub Secrets, and commits the encrypted backup to a private
   560→repository. This approach provides 7-day retention without additional
   561→cost. The encryption key is managed separately from the backup storage,
   562→ensuring that even if the repository is compromised, the backup data
   563→remains unreadable without the key. On migration to Supabase Pro, the
   564→daily automated backups with 7-day Point-In-Time Recovery become the
   565→primary mechanism, and the GitHub Actions backup is retained as a
   566→secondary off-site copy.
   567→
   568→[]{#_Toc100014 .anchor}**2.6.3 Subscription Gating**
   569→
   570→Access control and tier-based feature restrictions are enforced via
   571→Next.js Middleware, which runs on Vercel\'s edge network before any page
   572→renders. The middleware extracts the Supabase Auth JWT from the request
   573→cookie, decodes the user\'s plan metadata (stored as a custom claim in
   574→the JWT), and checks the current prescription count against the plan
   575→limit. If a Free-tier user has reached their 50-prescription monthly
   576→limit, the middleware redirects them to an upgrade prompt page rather
   577→than the prescription composer. Premium features such as custom
   578→letterhead branding and digital signature upload are gated by checking
   579→the plan claim before rendering the relevant UI components, and API
   580→routes perform a secondary server-side plan check to prevent client-side
   581→bypass.
   582→
   583→[]{#_Toc100015 .anchor}**3. Database Schema**
   584→
   585→The database is structured to support multi-tenancy where each doctor
   586→owns their specific settings and patient records. The schema uses
   587→Supabase PostgreSQL with Row Level Security (RLS) enforced on every
   588→table. All tables include standard audit columns (created_at,
   589→updated_at) and use UUID primary keys for global uniqueness and to
   590→prevent enumeration attacks.
   591→
   592→[]{#_Toc100016 .anchor}**3.1 Entity Relationship Overview**
   593→
   594→The core data model revolves around the doctor entity as the tenant
   595→root. Each doctor owns their settings, patients, and prescriptions. A
   596→patient belongs to exactly one doctor (to maintain data isolation), and
   597→a prescription belongs to one patient and one doctor. The subscriptions
   598→table tracks the doctor\'s current plan and usage counters, while the
   599→medicine_feedback table supports crowdsourced improvement of the
   600→medicine database.

   951→[]{#_Toc100025 .anchor}**3.3 Row Level Security Policies**
   952→
   953→Every table in the database is protected by RLS policies ensuring that a
   954→logged-in doctor can only select, insert, or update data where the
   955→doctor_id matches their authenticated session UID. The RLS policies are
   956→defined as follows for each table, using the auth.uid() function to
   957→extract the current user\'s ID from the JWT token. For the doctors
   958→table, the policy verifies that the id column matches auth.uid(). For
   959→all other tables with a doctor_id column, the policy checks that
   960→doctor_id equals auth.uid(). The audit_log table is write-only for
   961→doctors (INSERT allowed, SELECT restricted to admin roles) to prevent
   962→tampering with activity records.
   963→
   964→Additionally, RLS policies enforce subscription limits on the
   965→prescriptions table. An INSERT policy on prescriptions checks the
   966→subscriptions table to verify that the doctor has not exceeded their
   967→monthly prescription limit before allowing a new prescription to be
   968→created. This database-level enforcement prevents bypass through API
   969→route vulnerabilities or direct Supabase client access.
   970→
   971→[]{#_Toc100026 .anchor}**3.4 Indexes**
   972→
   973→The following indexes are created to optimize the most frequent query
   974→patterns. An index on patients(doctor_id, name) supports the patient
   975→search dropdown in the prescription composer. An index on
   976→prescriptions(doctor_id, created_at DESC) enables efficient history
   977→retrieval sorted by most recent visit. An index on
   978→prescriptions(patient_id, created_at DESC) supports the patient history
   979→view. An index on subscriptions(doctor_id) provides fast lookups for
   980→subscription gating middleware. A GIN index on
   981→prescriptions(medications) allows JSONB queries for analytics (e.g.,
   982→most prescribed medications). An index on medicine_feedback(status)
   983→supports the admin review queue for pending feedback items.
   984→
   985→[]{#_Toc100027 .anchor}**3.5 Database Migrations**
   986→
   987→Schema migrations are managed through Supabase\'s built-in migration
   988→system, which tracks migration files in a supabase/migrations/ directory
   989→within the project repository. Each migration is an up-only SQL file
   990→with a timestamp prefix (e.g., 20260614000000_create_subscriptions.sql).
   991→The supabase CLI is used to apply migrations to local development
   992→instances and to push changes to the remote Supabase project. Rollback
   993→is handled by creating a new migration that reverses the previous
   994→change, rather than using down migrations, ensuring a consistent and
   995→auditable migration history.

   997→[]{#_Toc100028 .anchor}**4. Medicine Data Strategy**
   998→
   999→RxBD employs a hybrid static-dynamic medicine data strategy that
  1000→prioritizes search speed and zero API latency while maintaining a clear
  1001→path for data updates. This approach recognizes that the Bangladesh
  1002→pharmaceutical market, while containing approximately 8,000 registered
  1003→brand-name products across 300+ generic molecules, is small enough to be
  1004→indexed entirely in client memory, yet large enough that manual curation
  1005→is impractical.
  1006→
  1007→[]{#_Toc100029 .anchor}**4.1 Data Source**
  1008→
  1009→The initial medicine database is sourced from the Directorate General of
  1010→Drug Administration (DGDA) of Bangladesh, which publishes a regularly
  1011→updated registry of all approved pharmaceutical products. This registry
  1012→includes brand names, generic compositions, dosage forms, strengths,
  1013→manufacturers, and retail prices. The raw DGDA data is processed through
  1014→an ETL (Extract, Transform, Load) pipeline that normalizes brand name
  1015→spellings, maps each brand to its generic molecule, and structures the
  1016→data into the JSON format required by Fuse.js. The pipeline is
  1017→version-controlled in the repository and can be re-run whenever the DGDA
  1018→publishes updated data.
  1019→
