     1→   530→health endpoint. Alert channels include email and Slack integration.
     2→   531→This provides basic availability monitoring sufficient for the launch
     3→   532→phase, with migration to Vercel\'s built-in analytics and Synthetics on
     4→   533→the Pro plan as the platform scales.
     5→   534→
     6→   535→[]{#_Toc100011 .anchor}**2.6 Data Resilience**
     7→   536→
     8→   537→Data resilience in RxBD operates at three levels, ensuring that no
     9→   538→single point of failure can result in permanent data loss.
    10→   539→
    11→   540→[]{#_Toc100012 .anchor}**2.6.1 Local-First Auto-Save**
    12→   541→
    13→   542→Prescription drafts are automatically persisted to the browser\'s
    14→   543→localStorage every 10 seconds using a debounced Zustand middleware. The
    15→   544→auto-save captures the full prescription state including patient
    16→   545→context, medications added, clinical notes, and form dirty state. Upon
    17→   546→reconnection after a connectivity loss, the application detects local
    18→   547→drafts by comparing the localStorage timestamp against the server\'s
    19→   548→last-known save time and prompts the user to resume their work. This
    20→   549→mechanism prevents data loss even during extended internet outages,
    21→   550→which are common in semi-urban Bangladesh, and requires zero server
    22→   551→infrastructure.
    23→   552→
    24→   553→[]{#_Toc100013 .anchor}**2.6.2 Database Backup Strategy**
    25→   554→
    26→   555→The primary backup mechanism uses Supabase\'s built-in daily automated
    27→   556→backups, which are available on the Pro plan. During the free-tier
    28→   557→launch phase, a GitHub Actions workflow performs a daily pg_dump of the
    29→   558→Supabase database, encrypts the dump using AES-256 encryption with a key
    30→   559→stored in GitHub Secrets, and commits the encrypted backup to a private
    31→   560→repository. This approach provides 7-day retention without additional
    32→   561→cost. The encryption key is managed separately from the backup storage,
    33→   562→ensuring that even if the repository is compromised, the backup data
    34→   563→remains unreadable without the key. On migration to Supabase Pro, the
    35→   564→daily automated backups with 7-day Point-In-Time Recovery become the
    36→   565→primary mechanism, and the GitHub Actions backup is retained as a
    37→   566→secondary off-site copy.
    38→   567→
    39→   568→[]{#_Toc100014 .anchor}**2.6.3 Subscription Gating**
    40→   569→
    41→   570→Access control and tier-based feature restrictions are enforced via
    42→   571→Next.js Middleware, which runs on Vercel\'s edge network before any page
    43→   572→renders. The middleware extracts the Supabase Auth JWT from the request
    44→   573→cookie, decodes the user\'s plan metadata (stored as a custom claim in
    45→   574→the JWT), and checks the current prescription count against the plan
    46→   575→limit. If a Free-tier user has reached their 50-prescription monthly
    47→   576→limit, the middleware redirects them to an upgrade prompt page rather
    48→   577→than the prescription composer. Premium features such as custom
    49→   578→letterhead branding and digital signature upload are gated by checking
    50→   579→the plan claim before rendering the relevant UI components, and API
    51→   580→routes perform a secondary server-side plan check to prevent client-side
    52→   581→bypass.
    53→   582→
    54→   583→[]{#_Toc100015 .anchor}**3. Database Schema**
    55→   584→
    56→   585→The database is structured to support multi-tenancy where each doctor
    57→   586→owns their specific settings and patient records. The schema uses
    58→   587→Supabase PostgreSQL with Row Level Security (RLS) enforced on every
    59→   588→table. All tables include standard audit columns (created_at,
    60→   589→updated_at) and use UUID primary keys for global uniqueness and to
    61→   590→prevent enumeration attacks.
    62→   591→
    63→   592→[]{#_Toc100016 .anchor}**3.1 Entity Relationship Overview**
    64→   593→
    65→   594→The core data model revolves around the doctor entity as the tenant
    66→   595→root. Each doctor owns their settings, patients, and prescriptions. A
    67→   596→patient belongs to exactly one doctor (to maintain data isolation), and
    68→   597→a prescription belongs to one patient and one doctor. The subscriptions
    69→   598→table tracks the doctor\'s current plan and usage counters, while the
    70→   599→medicine_feedback table supports crowdsourced improvement of the
    71→   600→medicine database.

   422→   951→[]{#_Toc100025 .anchor}**3.3 Row Level Security Policies**
   423→   952→
   424→   953→Every table in the database is protected by RLS policies ensuring that a
   425→   954→logged-in doctor can only select, insert, or update data where the
   426→   955→doctor_id matches their authenticated session UID. The RLS policies are
   427→   956→defined as follows for each table, using the auth.uid() function to
   428→   957→extract the current user\'s ID from the JWT token. For the doctors
   429→   958→table, the policy verifies that the id column matches auth.uid(). For
   430→   959→all other tables with a doctor_id column, the policy checks that
   431→   960→doctor_id equals auth.uid(). The audit_log table is write-only for
   432→   961→doctors (INSERT allowed, SELECT restricted to admin roles) to prevent
   433→   962→tampering with activity records.
   434→   963→
   435→   964→Additionally, RLS policies enforce subscription limits on the
   436→   965→prescriptions table. An INSERT policy on prescriptions checks the
   437→   966→subscriptions table to verify that the doctor has not exceeded their
   438→   967→monthly prescription limit before allowing a new prescription to be
   439→   968→created. This database-level enforcement prevents bypass through API
   440→   969→route vulnerabilities or direct Supabase client access.
   441→   970→
   442→   971→[]{#_Toc100026 .anchor}**3.4 Indexes**
   443→   972→
   444→   973→The following indexes are created to optimize the most frequent query
   445→   974→patterns. An index on patients(doctor_id, name) supports the patient
   446→   975→search dropdown in the prescription composer. An index on
   447→   976→prescriptions(doctor_id, created_at DESC) enables efficient history
   448→   977→retrieval sorted by most recent visit. An index on
   449→   978→prescriptions(patient_id, created_at DESC) supports the patient history
   450→   979→view. An index on subscriptions(doctor_id) provides fast lookups for
   451→   980→subscription gating middleware. A GIN index on
   452→   981→prescriptions(medications) allows JSONB queries for analytics (e.g.,
   453→   982→most prescribed medications). An index on medicine_feedback(status)
   454→   983→supports the admin review queue for pending feedback items.
   455→   984→
   456→   985→[]{#_Toc100027 .anchor}**3.5 Database Migrations**
   457→   986→
   458→   987→Schema migrations are managed through Supabase\'s built-in migration
   459→   988→system, which tracks migration files in a supabase/migrations/ directory
   460→   989→within the project repository. Each migration is an up-only SQL file
   461→   990→with a timestamp prefix (e.g., 20260614000000_create_subscriptions.sql).
   462→   991→The supabase CLI is used to apply migrations to local development
   463→   992→instances and to push changes to the remote Supabase project. Rollback
   464→   993→is handled by creating a new migration that reverses the previous
   465→   994→change, rather than using down migrations, ensuring a consistent and
   466→   995→auditable migration history.
   467→   996→
   468→   997→[]{#_Toc100028 .anchor}**4. Medicine Data Strategy**
   469→   998→
   470→   999→RxBD employs a hybrid static-dynamic medicine data strategy that
   471→  1000→prioritizes search speed and zero API latency while maintaining a clear
   472→  1001→path for data updates. This approach recognizes that the Bangladesh
   473→  1002→pharmaceutical market, while containing approximately 8,000 registered
   474→  1003→brand-name products across 300+ generic molecules, is small enough to be
   475→  1004→indexed entirely in client memory, yet large enough that manual curation
   476→  1005→is impractical.
   477→  1006→
   478→  1007→[]{#_Toc100029 .anchor}**4.1 Data Source**
   479→  1008→
   480→  1009→The initial medicine database is sourced from the Directorate General of
   481→  1010→Drug Administration (DGDA) of Bangladesh, which publishes a regularly
   482→  1011→updated registry of all approved pharmaceutical products. This registry
   483→  1012→includes brand names, generic compositions, dosage forms, strengths,
   484→  1013→manufacturers, and retail prices. The raw DGDA data is processed through
   485→  1014→an ETL (Extract, Transform, Load) pipeline that normalizes brand name
   486→  1015→spellings, maps each brand to its generic molecule, and structures the
   487→  1016→data into the JSON format required by Fuse.js. The pipeline is
   488→  1017→version-controlled in the repository and can be re-run whenever the DGDA
