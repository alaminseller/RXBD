   204→   204→[]{#_Toc100000 .anchor}**1. Executive Summary**
   205→   205→
   206→   206→RxBD is a comprehensive multi-tenant, SaaS-based digital prescription
   207→   207→platform purpose-built for private chamber doctors in Bangladesh. The
   208→   208→platform replaces error-prone handwritten prescriptions with a fast,
   209→   209→intuitive digital composer that generates professional, print-ready
   210→   210→prescriptions in under two minutes. By leveraging modern web
   211→   211→technologies and a free-tier infrastructure strategy centered on Vercel
   212→   212→and Supabase, RxBD achieves near-zero operational costs during its
   213→   213→initial growth phase while maintaining enterprise-grade security and
   214→   214→reliability.
   215→   215→
   216→   216→The Bangladesh healthcare market presents a significant opportunity for
   217→   217→digital prescription tools. The country has over 75,000 registered
   218→   218→physicians, with a substantial proportion operating private chambers
   219→   219→where prescription writing constitutes the primary clinical
   220→   220→documentation activity. Currently, the vast majority of these
   221→   221→practitioners rely on handwritten prescriptions, which are prone to
   222→   222→illegibility, incomplete drug information, and lack of verifiable
   223→   223→authenticity. Existing digital solutions such as Practo and local EMR
   224→   224→systems are either too complex, too expensive, or poorly adapted to the
   225→   225→specific workflow of a Bangladeshi chamber doctor who sees 30 to 50
   226→   226→patients per session and needs speed above all else.
   227→   227→
   228→   228→RxBD addresses this gap with a focused, opinionated product that
   229→   229→prioritizes speed of prescription creation, professional output quality,
   230→   230→and zero-friction onboarding. The platform employs a client-side
   231→   231→medicine search engine powered by Fuse.js over static JSON datasets,
   232→   232→eliminating API latency and reducing infrastructure costs. A local-first
   233→   233→auto-save mechanism ensures that no prescription draft is ever lost,
   234→   234→even during internet outages common in semi-urban and rural Bangladesh.
   235→   235→Every generated prescription includes a unique QR code for document
   236→   236→verification, building trust and combating prescription forgery.
   237→   237→
   238→   238→The commercial model follows a freemium SaaS approach: the Free tier
   239→   239→provides core functionality with a limit of 50 prescriptions per month,
   240→   240→sufficient for part-time practitioners, while the Premium tier unlocks
   241→   241→unlimited prescriptions, custom letterhead branding, digital signature
   242→   242→embedding, and priority support. Payment processing is handled through
   243→   243→SSLCommerz, supporting local mobile financial services such as bKash,
   244→   244→Nagad, and Rocket, ensuring accessibility for the target demographic.
   245→   245→
   246→   246→Infrastructure costs are managed through a deliberate free-tier
   247→   247→strategy. Vercel\'s hobby plan provides serverless hosting with 100GB
   248→   248→monthly bandwidth and 10-second function timeouts, while Supabase\'s
   249→   249→free tier offers 500MB of PostgreSQL storage, 1GB of file storage, and
   250→   250→50,000 monthly active users for authentication. These limits are
   251→   251→sufficient to support the initial user base of 500 to 1,000 doctors,
   252→   252→with a clear migration path to paid tiers as revenue grows. Automated,
   253→   253→encrypted database backups via Supabase\'s built-in Point-In-Time
   254→   254→Recovery and GitHub Actions provide data resilience without additional
   255→   255→cost.

   448→   448→[]{#_Toc100005 .anchor}**2.3 Architecture Overview**
   449→   449→
   450→   450→RxBD follows a flat, serverless architecture optimized for the JAMstack
   451→   451→paradigm. The entire application is deployed as a single Next.js project
   452→   452→on Vercel, with serverless API routes handling all backend logic and
   453→   453→Supabase providing managed database, authentication, and file storage
   454→   454→services. This architecture eliminates the need for container
   455→   455→orchestration, server provisioning, or infrastructure management,
   456→   456→allowing the team to focus entirely on product development.
   457→   457→
   458→   458→The request flow follows a clear pattern: client requests hit Vercel\'s
   459→   459→edge CDN first, where static assets are served from cache. Dynamic
   460→   460→routes are handled by Next.js serverless functions, which authenticate
   461→   461→requests via Supabase Auth JWT tokens, enforce Row Level Security
   462→   462→through Supabase\'s PostgreSQL policies, and return data. The
   463→   463→prescription composer operates as a client-side application that loads
   464→   464→medicine data into memory once on page load, enabling instant fuzzy
   465→   465→search without server round-trips.
   466→   466→
   467→   467→Multi-tenant data isolation is enforced at the database level through
   468→   468→Supabase Row Level Security (RLS). Every table includes a doctor_id
   469→   469→foreign key referencing the auth.users table, and RLS policies ensure
   470→   470→that each authenticated doctor can only access rows where the doctor_id
   471→   471→matches their session UID. This approach is both simpler and more secure
   472→   472→than application-level tenant filtering, as it prevents accidental data
   473→   473→leakage even if a developer introduces a bug in a query.
   474→   474→
   475→   475→[]{#_Toc100006 .anchor}**2.4 Architecture Diagram (Textual
   476→   476→Description)**
   477→   477→
   478→   478→The system architecture can be described in four layers. Layer 1
   479→   479→(Client) consists of the Next.js frontend application running in the
   480→   480→user\'s browser, including the prescription composer with Zustand state
   481→   481→management, Fuse.js medicine search index loaded from static JSON, and
   482→   482→\@react-pdf/renderer for client-side PDF generation. Layer 2 (Edge/CDN)
   483→   483→is Vercel\'s global edge network, serving cached static assets and
   484→   484→routing dynamic requests to serverless functions. Layer 3 (API)
   485→   485→comprises Next.js API routes running as serverless functions, handling
   486→   486→authentication validation, database CRUD operations, subscription
   487→   487→enforcement, and webhook processing for SSLCommerz payment
   488→   488→confirmations. Layer 4 (Data) includes Supabase PostgreSQL for
   489→   489→persistent data with RLS, Supabase Auth for user management, Supabase
   490→   490→Storage for doctor-uploaded assets, and the browser\'s localStorage for
   491→   491→draft auto-save persistence.

   500→   500→[]{#_Toc100008 .anchor}**2.5.1 Error Tracking**

   503→   503→Sentry Next.js SDK captures unhandled exceptions, React component

   507→   507→doctors. Error events are tagged with the doctor\'s subscription tier

   509→   509→defined as any error in the prescription creation or PDF generation
   510→   510→flow, trigger immediate Slack notifications via Sentry\'s integration
   511→   511→webhooks.

   535→   535→[]{#_Toc100011 .anchor}**2.6 Data Resilience**
   536→   536→
   537→   537→Data resilience in RxBD operates at three levels, ensuring that no
   538→   538→single point of failure can result in permanent data loss.
   539→   539→
   540→   540→[]{#_Toc100012 .anchor}**2.6.1 Local-First Auto-Save**
   541→   541→
   542→   542→Prescription drafts are automatically persisted to the browser\'s
   543→   543→localStorage every 10 seconds using a debounced Zustand middleware. The
   544→   544→auto-save captures the full prescription state including patient
   545→   545→context, medications added, clinical notes, and form dirty state. Upon
   546→   546→reconnection after a connectivity loss, the application detects local
   547→   547→drafts by comparing the localStorage timestamp against the server\'s
   548→   548→last-known save time and prompts the user to resume their work. This
   549→   549→mechanism prevents data loss even during extended internet outages,
   550→   550→which are common in semi-urban Bangladesh, and requires zero server
   551→   551→infrastructure.
   552→   552→
   553→   553→[]{#_Toc100013 .anchor}**2.6.2 Database Backup Strategy**
   554→   554→
   555→   555→The primary backup mechanism uses Supabase\'s built-in daily automated
   556→   556→backups, which are available on the Pro plan. During the free-tier
   557→   557→launch phase, a GitHub Actions workflow performs a daily pg_dump of the
   558→   558→Supabase database, encrypts the dump using AES-256 encryption with a key
   559→   559→stored in GitHub Secrets, and commits the encrypted backup to a private
   560→   560→repository. This approach provides 7-day retention without additional
   561→   561→cost. The encryption key is managed separately from the backup storage,
   562→   562→ensuring that even if the repository is compromised, the backup data
   563→   563→remains unreadable without the key. On migration to Supabase Pro, the
   564→   564→daily automated backups with 7-day Point-In-Time Recovery become the
   565→   565→primary mechanism, and the GitHub Actions backup is retained as a
   566→   566→secondary off-site copy.
   567→   567→
   568→   568→[]{#_Toc100014 .anchor}**2.6.3 Subscription Gating**
   569→   569→
   570→   570→Access control and tier-based feature restrictions are enforced via
   571→   571→Next.js Middleware, which runs on Vercel\'s edge network before any page
   572→   572→renders. The middleware extracts the Supabase Auth JWT from the request
   573→   573→cookie, decodes the user\'s plan metadata (stored as a custom claim in
   574→   574→the JWT), and checks the current prescription count against the plan
   575→   575→limit. If a Free-tier user has reached their 50-prescription monthly
   576→   576→limit, the middleware redirects them to an upgrade prompt page rather
   577→   577→than the prescription composer. Premium features such as custom
   578→   578→letterhead branding and digital signature upload are gated by checking
   579→   579→the plan claim before rendering the relevant UI components, and API
   580→   580→routes perform a secondary server-side plan check to prevent client-side
   581→   581→bypass.
   582→   582→
   583→   583→[]{#_Toc100015 .anchor}**3. Database Schema**
   584→   584→
   585→   585→The database is structured to support multi-tenancy where each doctor
   586→   586→owns their specific settings and patient records. The schema uses
   587→   587→Supabase PostgreSQL with Row Level Security (RLS) enforced on every
   588→   588→table. All tables include standard audit columns (created_at,
   589→   589→updated_at) and use UUID primary keys for global uniqueness and to
   590→   590→prevent enumeration attacks.
   591→   591→
   592→   592→[]{#_Toc100016 .anchor}**3.1 Entity Relationship Overview**
   593→   593→
   594→   594→The core data model revolves around the doctor entity as the tenant
   595→   595→root. Each doctor owns their settings, patients, and prescriptions. A
   596→   596→patient belongs to exactly one doctor (to maintain data isolation), and
   597→   597→a prescription belongs to one patient and one doctor. The subscriptions
   598→   598→table tracks the doctor\'s current plan and usage counters, while the
   599→   599→medicine_feedback table supports crowdsourced improvement of the
   600→   600→medicine database.

   951→   951→[]{#_Toc100025 .anchor}**3.3 Row Level Security Policies**
   952→   952→
   953→   953→Every table in the database is protected by RLS policies ensuring that a
   954→   954→logged-in doctor can only select, insert, or update data where the
   955→   955→doctor_id matches their authenticated session UID. The RLS policies are
   956→   956→defined as follows for each table, using the auth.uid() function to
   957→   957→extract the current user\'s ID from the JWT token. For the doctors
   958→   958→table, the policy verifies that the id column matches auth.uid(). For
   959→   959→all other tables with a doctor_id column, the policy checks that
   960→   960→doctor_id equals auth.uid(). The audit_log table is write-only for
   961→   961→doctors (INSERT allowed, SELECT restricted to admin roles) to prevent
   962→   962→tampering with activity records.
   963→   963→
   964→   964→Additionally, RLS policies enforce subscription limits on the
   965→   965→prescriptions table. An INSERT policy on prescriptions checks the
   966→   966→subscriptions table to verify that the doctor has not exceeded their
   967→   967→monthly prescription limit before allowing a new prescription to be
   968→   968→created. This database-level enforcement prevents bypass through API
   969→   969→route vulnerabilities or direct Supabase client access.
   970→   970→
   971→   971→[]{#_Toc100026 .anchor}**3.4 Indexes**
   972→   972→
   973→   973→The following indexes are created to optimize the most frequent query
   974→   974→patterns. An index on patients(doctor_id, name) supports the patient
   975→   975→search dropdown in the prescription composer. An index on
   976→   976→prescriptions(doctor_id, created_at DESC) enables efficient history
   977→   977→retrieval sorted by most recent visit. An index on
   978→   978→prescriptions(patient_id, created_at DESC) supports the patient history
   979→   979→view. An index on subscriptions(doctor_id) provides fast lookups for
   980→   980→subscription gating middleware. A GIN index on
   981→   981→prescriptions(medications) allows JSONB queries for analytics (e.g.,
   982→   982→most prescribed medications). An index on medicine_feedback(status)
   983→   983→supports the admin review queue for pending feedback items.
   984→   984→
   985→   985→[]{#_Toc100027 .anchor}**3.5 Database Migrations**
   986→   986→
   987→   987→Schema migrations are managed through Supabase\'s built-in migration
   988→   988→system, which tracks migration files in a supabase/migrations/ directory
   989→   989→within the project repository. Each migration is an up-only SQL file
   990→   990→with a timestamp prefix (e.g., 20260614000000_create_subscriptions.sql).
   991→   991→The supabase CLI is used to apply migrations to local development
   992→   992→instances and to push changes to the remote Supabase project. Rollback
   993→   993→is handled by creating a new migration that reverses the previous
   994→   994→change, rather than using down migrations, ensuring a consistent and
   995→   995→auditable migration history.

   997→   997→[]{#_Toc100028 .anchor}**4. Medicine Data Strategy**
   998→   998→
   999→   999→RxBD employs a hybrid static-dynamic medicine data strategy that
  1000→  1000→prioritizes search speed and zero API latency while maintaining a clear
  1001→  1001→path for data updates. This approach recognizes that the Bangladesh
  1002→  1002→pharmaceutical market, while containing approximately 8,000 registered
  1003→  1003→brand-name products across 300+ generic molecules, is small enough to be
  1004→  1004→indexed entirely in client memory, yet large enough that manual curation
  1005→  1005→is impractical.
  1006→  1006→
  1007→  1007→[]{#_Toc100029 .anchor}**4.1 Data Source**
  1008→  1008→
  1009→  1009→The initial medicine database is sourced from the Directorate General of
  1010→  1010→Drug Administration (DGDA) of Bangladesh, which publishes a regularly
  1011→  1011→updated registry of all approved pharmaceutical products. This registry
  1012→  1012→includes brand names, generic compositions, dosage forms, strengths,
  1013→  1013→manufacturers, and retail prices. The raw DGDA data is processed through
  1014→  1014→an ETL (Extract, Transform, Load) pipeline that normalizes brand name
  1015→  1015→spellings, maps each brand to its generic molecule, and structures the
  1016→  1016→data into the JSON format required by Fuse.js. The pipeline is
  1017→  1017→version-controlled in the repository and can be re-run whenever the DGDA
  1018→  1018→publishes updated data.
  1019→  1019→
  1020→  1020→[]{#_Toc100030 .anchor}**4.2 Static JSON Dataset Architecture**
  1021→  1021→
  1022→  1022→Medicine data is stored as optimized static JSON files in the
  1023→  1023→/public/data/ directory. The primary file, medicines.json, contains an
  1024→  1024→array of objects, each with the following structure: { id, brand,
  1025→  1025→generic, dosageForm, strength, manufacturer, category, tags }. A
  1026→  1026→secondary file, generics.json, provides a mapping of generic molecule
  1027→  1027→names to their pharmacological categories, common side effects, and
  1028→  1028→typical dosage ranges. Both files are loaded once when the prescription
  1029→  1029→composer page mounts, and Fuse.js creates in-memory indices over the
  1030→  1030→brand, generic, and tags fields.
  1031→  1031→
  1032→  1032→The estimated file size for medicines.json is approximately 2MB for
  1033→  1033→8,000 entries, which compresses to under 400KB with Vercel\'s automatic
  1034→  1034→Brotli compression. This size is well within acceptable limits for a
  1035→  1035→one-time page-load fetch and has negligible impact on the user\'s
  1036→  1036→initial bandwidth, even on slow mobile connections common in Bangladesh.
  1037→  1037→
  1038→  1038→[]{#_Toc100031 .anchor}**4.3 Search Behavior**
  1039→  1039→
  1040→  1040→Fuse.js is configured with a threshold of 0.3 (balancing precision and
  1041→  1041→recall), keys indexed on brand (weight 2), generic (weight 1.5), and
  1042→  1042→tags (weight 1). This configuration ensures that a partial brand name
  1043→  1043→like \'amo\' returns \'Amoxicillin\' brands as top results, while a
  1044→  1044→generic search for \'paracetamol\' returns all paracetamol-containing
  1045→  1045→brands. Results are displayed in a dropdown with the brand name
  1046→  1046→highlighted and the generic composition shown as a secondary line,
  1047→  1047→enabling doctors to quickly identify the correct medication even when
  1048→  1048→multiple brands share similar names.
  1049→  1049→
  1050→  1050→[]{#_Toc100032 .anchor}**4.4 Data Update Pipeline**
  1051→  1051→
  1052→  1052→Updates to the medicine database follow a two-tier strategy. Tier 1 (Hot
  1053→  1053→Updates) uses a Supabase table called medicine_hotfixes that stores
  1054→  1054→corrections and additions submitted through the crowdsourced feedback
  1055→  1055→system. When the prescription composer loads, it fetches any hotfixes
  1056→  1056→created since the last JSON deployment date and merges them into the
  1057→  1057→in-memory Fuse.js index. This ensures that critical corrections (e.g., a
  1058→  1058→dangerous dosage error) are available within minutes without requiring a
  1059→  1059→full deployment. Tier 2 (Cold Updates) involves periodic re-runs of the
  1060→  1060→DGDA ETL pipeline, regeneration of the static JSON files, and deployment
  1061→  1061→of a new version via Vercel. This happens on a monthly cadence or when
  1062→  1062→the hotfix table accumulates more than 50 entries, at which point the
  1063→  1063→hotfixes are folded into the main JSON dataset.
  1064→  1064→
  1065→  1065→[]{#_Toc100033 .anchor}**4.5 Crowdsourced Feedback Workflow**
  1066→  1066→
  1067→  1067→The prescription composer includes a feedback button that allows doctors
  1068→  1068→to flag incorrect or missing medicine data. When clicked, it opens a
  1069→  1069→lightweight form pre-populated with the current medicine name and a
  1070→  1070→dropdown for feedback type (missing, incorrect, duplicate, suggestion).
  1071→  1071→The submission is stored in the medicine_feedback table with a status of
  1072→  1072→\'pending\'. An admin dashboard, accessible only to the RxBD operations
  1073→  1073→team, displays pending feedback items sorted by submission date.
  1074→  1074→Reviewed items are either resolved (by creating a hotfix entry in the
  1075→  1075→medicine_hotfixes table) or rejected (with a reason recorded in the
  1076→  1076→audit_log). The submitting doctor receives a notification when their
  1077→  1077→feedback is resolved, encouraging continued participation in the
  1078→  1078→improvement process.

  1124→  1124→[]{#_Toc100036 .anchor}**5.2 Monthly Prescription Counter Mechanism**
  1125→  1125→
  1126→  1126→The 50-prescription monthly limit for Free-tier users is enforced
  1127→  1127→through a combination of database-level and application-level checks.
  1128→  1128→The subscriptions table maintains a prescriptions_this_month counter and
  1129→  1129→a counter_reset_at date. Every time a Free-tier doctor creates a new
  1130→  1130→prescription (transitions from draft to final), the API route increments
  1131→  1131→the counter. Before allowing the creation, it checks whether the counter
  1132→  1132→has reached the 50-prescription limit. If counter_reset_at is older than
  1133→  1133→30 days, the counter is reset to 1 and the date is updated to the
  1134→  1134→current date.
  1135→  1135→
  1136→  1136→A scheduled Supabase Edge Function runs daily at midnight Bangladesh
  1137→  1137→time (UTC+6) to reset all Free-tier counters where counter_reset_at is
  1138→  1138→older than 30 days. This ensures that the counter resets even if a
  1139→  1139→doctor does not visit the platform on their reset date. Additionally,
  1140→  1140→the RLS policy on the prescriptions table includes a subquery check
  1141→  1141→against the subscriptions table, providing a secondary enforcement layer
  1142→  1142→that prevents counter bypass through direct database access.
  1143→  1143→
  1144→  1144→[]{#_Toc100037 .anchor}**5.3 Free-to-Premium Upgrade Flow**
  1145→  1145→
  1146→  1146→When a Free-tier doctor approaches their monthly limit (at 40
  1147→  1147→prescriptions), a non-intrusive banner appears at the top of the
  1148→  1148→prescription composer reading \'You have 10 prescriptions remaining this
  1149→  1149→month. Upgrade to Premium for unlimited access.\' At 48 prescriptions,
  1150→  1150→the banner becomes more prominent with a yellow highlight. At 50
  1151→  1151→prescriptions, the composer is disabled and replaced with a full-screen
  1152→  1152→upgrade prompt that explains the Premium benefits and includes a direct
  1153→  1153→\'Upgrade Now\' button. This graduated approach provides multiple
  1154→  1154→touchpoints for conversion without alienating users with hard paywalls
  1155→  1155→at the first sign of friction.
  1156→  1156→
  1157→  1157→The upgrade flow itself redirects the doctor to a Stripe Checkout-like
  1158→  1158→page powered by SSLCommerz. After successful payment, a webhook from
  1159→  1159→SSLCommerz hits the /api/webhooks/sslcommerz API route, which verifies
  1160→  1160→the payment signature using the store\'s secret key, updates the
  1161→  1161→subscriptions table to set plan=\'premium\', records the SSLCommerz
  1162→  1162→transaction ID, and sets the premium_expires_at date. The doctor\'s JWT
  1163→  1163→is refreshed on their next request to include the updated plan claim,
  1164→  1164→immediately unlocking Premium features.
  1165→  1165→
  1166→  1166→[]{#_Toc100038 .anchor}**5.4 Payment Integration via SSLCommerz**
  1167→  1167→
  1168→  1168→SSLCommerz is the leading payment gateway in Bangladesh, supporting all
  1169→  1169→local payment methods including bKash, Nagad, Rocket, DBBL Nexus, and
  1170→  1170→international Visa/Mastercard. Integration follows SSLCommerz\'s hosted
  1171→  1171→checkout flow: the RxBD backend generates a payment session by calling
  1172→  1172→SSLCommerz\'s /gwprocess/v4/api.php endpoint with the doctor\'s
  1173→  1173→information and payment amount, receives a redirect URL, and sends the
  1174→  1174→doctor to SSLCommerz\'s hosted payment page. After payment completion,
  1175→  1175→SSLCommerz sends an IPN (Instant Payment Notification) webhook to the
  1176→  1176→/api/webhooks/sslcommerz endpoint, which is the authoritative source of
  1177→  1177→payment confirmation. The Premium tier is priced at BDT 499 per month
  1178→  1178→(approximately USD 4.50), a price point validated through interviews
  1179→  1179→with 15 pilot doctors who indicated willingness to pay between BDT 300
  1180→  1180→and BDT 800 for premium prescription features.
  1181→  1181→
  1182→  1182→[]{#_Toc100039 .anchor}**5.5 Downgrade and Cancellation Policy**
  1183→  1183→
  1184→  1184→When a Premium subscription expires (premium_expires_at passes without
  1185→  1185→renewal), the system automatically reverts the doctor to the Free tier.
  1186→  1186→The prescriptions_this_month counter is initialized to the count of
  1187→  1187→prescriptions created in the current calendar month, ensuring that the
  1188→  1188→doctor does not receive a fresh 50-prescription allocation upon
  1189→  1189→downgrade. Existing Premium features (custom letterhead, digital
  1190→  1190→signature) are preserved in the doctor_settings table but are no longer
  1191→  1191→rendered in the PDF output. The doctor\'s data, including all
  1192→  1192→prescription history, remains fully accessible. A 3-day grace period
  1193→  1193→after expiration allows continued Premium access while a renewal
  1194→  1194→reminder is sent via email and in-app notification.

  1392→  1392→[]{#_Toc100048 .anchor}**7.1 Composer Layout (Desktop)**
  1393→  1393→
  1394→  1394→The prescription composer uses a three-panel split-screen layout
  1395→  1395→optimized for the clinical workflow of a chamber doctor. The Left
  1396→  1396→Sidebar (25% width) displays the patient context: selected patient
  1397→  1397→demographics, persistent medical notes (allergies, chronic conditions),
  1398→  1398→and quick access to the last 5 prescriptions for returning patients. The
  1399→  1399→Center Workspace (50% width) is the Rx Pad, containing the multi-section
  1400→  1400→prescription form arranged in clinical order: Chief Complaints, Vitals,
  1401→  1401→On Examination, Investigations, Diagnosis, Medications, Advice, and
  1402→  1402→Follow-Up. Each section is collapsible to reduce visual clutter. The
  1403→  1403→Right Sidebar (25% width) provides the medicine search input at the top,
  1404→  1404→followed by a diagnosis template picker and a list of the doctor\'s
  1405→  1405→frequently prescribed medications for quick access.
  1406→  1406→
  1407→  1407→[]{#_Toc100049 .anchor}**7.2 Composer Layout (Mobile & Tablet)**
  1408→  1408→
  1409→  1409→On screens narrower than 1024px, the three-panel layout transforms into
  1410→  1410→a tabbed interface to preserve all functionality without horizontal
  1411→  1411→scrolling. Three tabs are displayed at the bottom of the screen: Patient
  1412→  1412→(left sidebar content), Rx Pad (center workspace), and Search (right
  1413→  1413→sidebar content). The active tab takes the full screen width. The
  1414→  1414→medication search results appear as a modal overlay from the Search tab,
  1415→  1415→allowing the doctor to select a medication and be returned to the Rx Pad
  1416→  1416→tab with the medication pre-filled. On phones narrower than 640px, the
  1417→  1417→form sections within the Rx Pad are displayed as an accordion rather
  1418→  1418→than collapsible panels, ensuring that only one section is expanded at a
  1419→  1419→time to maximize usable screen space.

  1421→  1421→[]{#_Toc100050 .anchor}**7.3 Print Layout**
  1422→  1422→
  1423→  1423→The prescription print layout adheres to the standard Bangladesh medical
  1424→  1424→prescription format on A4 paper (210mm x 297mm). The header section
  1425→  1425→occupies the top 15% of the page, displaying the doctor\'s name (in
  1426→  1426→large bold font), degrees (comma-separated), specialty, chamber name and
  1427→  1427→address, and optional logo aligned to the right. The left column (30%
  1428→  1428→width) contains Chief Complaints, Vitals, On Examination, and
  1429→  1429→Investigations, each with a labeled heading and the entered data. The
  1430→  1430→main section (70% width) displays the Rx symbol followed by the
  1431→  1431→medications list, where each medication is formatted with the brand
  1432→  1432→name, generic name in parentheses (bold), dosage, frequency, duration,
  1433→  1433→and any special instructions on subsequent lines. The advice section and
  1434→  1434→follow-up date appear below the medications. The footer section contains
  1435→  1435→the doctor\'s signature (or uploaded digital signature image for
  1436→  1436→Premium), the prescription date, and the QR code (2cm x 2cm) in the
  1437→  1437→bottom-right corner.

  1439→  1439→[]{#_Toc100051 .anchor}**7.4 Onboarding & Trust**
  1440→  1440→
  1441→  1441→The onboarding experience is designed to achieve the \'Aha!\' moment
  1442→  1442→within 5 minutes of signup. A 3-step guided walkthrough leads the new
  1443→  1443→doctor through: Step 1 (Complete Your Profile) entering their name,
  1444→  1444→degrees, specialty, and chamber information; Step 2 (Customize Your
  1445→  1445→Prescription) uploading a logo and configuring the header layout with a
  1446→  1446→live preview; Step 3 (Write Your First Prescription) completing a test
  1447→  1447→prescription with a dummy patient to experience the full workflow. Each
  1448→  1448→step includes contextual tooltips and a progress indicator. The
  1449→  1449→walkthrough can be skipped and resumed later from the Settings page, but
  1450→  1450→completing it unlocks a \'Verified Doctor\' badge on their prescription
  1451→  1451→header.
  1452→  1452→
  1453→  1453→The QR code verification system serves as both a trust mechanism and an
  1454→  1454→anti-forgery safeguard. Every prescription PDF includes a unique QR code
  1455→  1455→in the footer that encodes the URL
  1456→  1456→rxbd.com.bd/verify/\[prescription_id\]. Scanning the QR code opens a
  1457→  1457→public, read-only verification page that displays the doctor\'s name,
  1458→  1458→patient\'s name, visit date, and a tamper-evident status indicator
  1459→  1459→(green checkmark for verified, red warning if the prescription has been
  1460→  1460→modified since issuance). The verification page is implemented as a
  1461→  1461→server-side rendered Next.js page with ISR (Incremental Static
  1462→  1462→Regeneration) for performance, using a 60-second revalidation period to
  1463→  1463→balance freshness with cache efficiency.

  1465→  1465→[]{#_Toc100052 .anchor}**7.5 Internationalization (i18n)**
  1466→  1466→
  1467→  1467→RxBD supports two languages: English (default) and Bengali (Bangla). The
  1468→  1468→interface language is determined first by the user\'s browser
  1469→  1469→Accept-Language header, then by the locale preference stored in the
  1470→  1470→doctor\'s profile, and finally by a manual toggle in the navigation bar.
  1471→  1471→All UI strings, form labels, button text, error messages, and onboarding
  1472→  1472→content are externalized into message catalogs using next-intl. The
  1473→  1473→prescription content itself (chief complaints, diagnosis, medication

  1477→  1477→the selected locale, adjusting for Bengali text direction
  1478→  1478→(left-to-right, same as English) and font rendering using Noto Sans
  1479→  1479→Bengali.

  1582→  1582→[]{#_Toc100054 .anchor}**9. Data Privacy & Compliance**
  1583→  1583→
  1584→  1584→As a platform handling sensitive health information, RxBD must comply
  1585→  1585→with Bangladesh\'s data protection regulations and adhere to
  1586→  1586→international best practices for health data security. This section
  1587→  1587→details the compliance framework, data handling policies, and technical
  1588→  1588→safeguards implemented to protect patient and doctor data.
  1589→  1589→
  1590→  1590→[]{#_Toc100055 .anchor}**9.1 Regulatory Compliance**
  1591→  1591→
  1592→  1592→RxBD\'s data handling practices are designed to comply with the
  1593→  1593→Bangladesh Digital Security Act 2018, which establishes legal
  1594→  1594→requirements for the collection, processing, and storage of digital
  1595→  1595→data, including provisions for data breach notification and penalties
  1596→  1596→for unauthorized data access. Additionally, the platform aligns with the
  1597→  1597→principles of the Bangladesh Personal Data Protection Bill (draft),
  1598→  1598→which, while not yet enacted, is expected to establish data subject
  1599→  1599→rights, consent requirements, and cross-border data transfer
  1600→  1600→restrictions. By proactively adhering to these anticipated requirements,
  1601→  1601→RxBD positions itself for compliance with minimal disruption when the
  1602→  1602→bill becomes law.
  1603→  1603→
  1604→  1604→The platform also follows the technical guidance provided by the
  1605→  1605→Bangladesh Medical and Dental Council (BMDC) regarding electronic
  1606→  1606→medical records, which recommends that digital prescription systems
  1607→  1607→maintain audit trails, enforce access controls, and ensure data
  1608→  1608→portability. While the BMDC has not yet published binding regulations
  1609→  1609→for digital prescriptions, RxBD\'s implementation of RLS, audit logging,
  1610→  1610→and CSV/JSON export functionality exceeds current advisory standards.
  1611→  1611→
  1612→  1612→[]{#_Toc100056 .anchor}**9.2 Data Residency**
  1613→  1613→
  1614→  1614→All patient and prescription data is stored exclusively in Supabase\'s
  1615→  1615→AWS ap-southeast-1 (Singapore) region, which is the closest available
  1616→  1616→Supabase region to Bangladesh and provides sub-50ms latency for most
  1617→  1617→Bangladeshi users. Supabase does not replicate data to other regions
  1618→  1618→without explicit customer configuration, ensuring that health data
  1619→  1619→remains within the Asia-Pacific jurisdiction. No data is transferred to
  1620→  1620→servers in the United States, European Union, or any other jurisdiction
  1621→  1621→with differing data protection standards. The choice of Singapore as the
  1622→  1622→data residency location is documented in the platform\'s privacy policy
  1623→  1623→and is non-negotiable, even if a different region would offer lower
  1624→  1624→costs.
  1625→  1625→
  1626→  1626→[]{#_Toc100057 .anchor}**9.3 Encryption**
  1627→  1627→
  1628→  1628→Data at rest is protected by Supabase\'s default AES-256 encryption for
  1629→  1629→all PostgreSQL data files and backups. Data in transit is encrypted via
  1630→  1630→TLS 1.3 for all connections between the client, Vercel, and Supabase.
  1631→  1631→Database backups generated by the GitHub Actions workflow are encrypted
  1632→  1632→using AES-256-CBC with a 256-bit key stored in GitHub Secrets (separate
  1633→  1633→from the backup repository). The encryption key is rotated every 90 days
  1634→  1634→by generating a new key, updating the GitHub Secret, and performing a
  1635→  1635→fresh encrypted backup before deleting backups encrypted with the old
  1636→  1636→key.
  1637→  1637→
  1638→  1638→[]{#_Toc100058 .anchor}**9.4 Data Retention & Deletion**
  1639→  1639→
  1640→  1640→When a doctor deletes their account, a 30-day grace period begins during
  1641→  1641→which the account can be restored by contacting support. During this
  1642→  1642→period, the doctor\'s data remains in the database but is inaccessible
  1643→  1643→(the RLS policy denies access to deleted users). After 30 days, an
  1644→  1644→automated Edge Function permanently deletes all data associated with the
  1645→  1645→doctor, including their profile, settings, patients, prescriptions, and
  1646→  1646→uploaded assets from Supabase Storage. The audit_log entries for the
  1647→  1647→deleted doctor are retained for 7 years with the doctor_id anonymized to
  1648→  1648→a hash, as required by the Digital Security Act 2018 for financial and
  1649→  1649→legal record-keeping purposes. Patients do not have direct accounts in
  1650→  1650→the system and cannot self-service delete their data; however, a doctor
  1651→  1651→can delete individual patient records, which are permanently removed
  1652→  1652→after the 30-day grace period.
  1653→  1653→
  1654→  1654→[]{#_Toc100059 .anchor}**9.5 Patient Consent Mechanism**
  1655→  1655→
  1656→  1656→When a doctor creates a new patient record, the patient entry form
  1657→  1657→includes a required checkbox: \'Patient has been informed that their
  1658→  1658→prescription data will be stored digitally on the RxBD platform.\' This
  1659→  1659→consent is recorded as a boolean field (consent_given) with a timestamp
  1660→  1660→on the patient record. While the current implementation relies on the
  1661→  1661→doctor to obtain verbal consent during the consultation, the future
  1662→  1662→Patient Portal feature (see Section 12) will enable direct digital
  1663→  1663→consent through a one-time SMS verification flow.
  1664→  1664→
  1665→  1665→[]{#_Toc100060 .anchor}**9.6 Data Portability**
  1666→  1666→
  1667→  1667→Doctors can export their complete patient list and prescription history
  1668→  1668→at any time through the Settings page. Free-tier doctors can export in
  1669→  1669→CSV format, which includes patient demographics and prescription
  1670→  1670→summaries (date, diagnosis, medication count). Premium doctors can
  1671→  1671→additionally export in JSON format, which includes the full prescription
  1672→  1672→payload including the medications JSONB array, enabling migration to
  1673→  1673→other EMR systems. The export is generated server-side as a streaming
  1674→  1674→response to handle large datasets without memory issues, and a download
  1675→  1675→link is emailed to the doctor upon completion.
  1676→  1676→
  1677→  1677→[]{#_Toc100061 .anchor}**9.7 Privacy Transparency**
  1678→  1678→
  1679→  1679→RxBD maintains a strict, non-negotiable policy against selling, sharing,
  1680→  1680→or monetizing patient or doctor data with third parties for any purpose.
  1681→  1681→This policy is enforced technically through RLS (no admin user has bulk
  1682→  1682→access to patient data), organizationally through access controls on the
  1683→  1683→Supabase dashboard (only the founding team has admin access), and
  1684→  1684→contractually through the Terms of Service. Anonymized, aggregate
  1685→  1685→analytics (e.g., total prescriptions created per day, most searched
  1686→  1686→generic names) may be used for product improvement but are never shared
  1687→  1687→externally.

  1830→  1830→[]{#_Toc100067 .anchor}**11. Operational Readiness**
  1831→  1831→
  1832→  1832→Operational readiness ensures that the RxBD team has the tools,
  1833→  1833→processes, and information needed to support doctors effectively from
  1834→  1834→day one. This section covers support infrastructure, beta testing
  1835→  1835→strategy, and analytics implementation.
  1836→  1836→
  1837→  1837→[]{#_Toc100068 .anchor}**11.1 Support Strategy**
  1838→  1838→
  1839→  1839→RxBD implements a tiered support model aligned with the subscription
  1840→  1840→tiers. Free-tier doctors receive support through a community forum
  1841→  1841→(built with GitHub Discussions for zero cost) and email support with a
  1842→  1842→72-hour response SLA. Premium doctors receive priority support via
  1843→  1843→WhatsApp Business API and email with a 24-hour response SLA.
  1844→  1844→
  1845→  1845→The WhatsApp Business API integration is implemented using the WhatsApp
  1846→  1846→Business Platform (Cloud API, free tier: 1,000 conversations per month).
  1847→  1847→A dedicated business number is configured through Meta\'s Business
  1848→  1848→Manager. When a Premium doctor sends a WhatsApp message to the RxBD
  1849→  1849→support number, the message is received via webhook, logged in a support
  1850→  1850→tracking spreadsheet, and routed to the on-call support team member.
  1851→  1851→Responses are sent through the same WhatsApp channel, maintaining a
  1852→  1852→conversational context. For the launch phase, the support team consists
  1853→  1853→of the founding members, with a plan to hire a dedicated support agent
  1854→  1854→once Premium subscriptions exceed 50 active users.
  1855→  1855→
  1856→  1856→[]{#_Toc100069 .anchor}**11.2 Beta Feedback Loop**
  1857→  1857→
  1858→  1858→The beta program targets 3 to 5 pilot doctors selected through direct
  1859→  1859→outreach to the founding team\'s professional network. Selection
  1860→  1860→criteria include: active private chamber practice with a minimum of 20
  1861→  1861→patients per day, willingness to provide structured feedback via a
  1862→  1862→dedicated Google Form after each session, representation of at least 2
  1863→  1863→different medical specialties (to test specialty-specific prescription
  1864→  1864→patterns), and varying levels of technical literacy (from tech-savvy to
  1865→  1865→minimal computer experience).
  1866→  1866→
  1867→  1867→Feedback is collected through three channels: a structured Google Form
  1868→  1868→distributed after each day of use (covering ease of use, search
  1869→  1869→accuracy, PDF quality, and feature requests), a dedicated WhatsApp group
  1870→  1870→for real-time issues and suggestions, and automated analytics tracking
  1871→  1871→of key usage metrics. The feedback is reviewed weekly by the founding
  1872→  1872→team, prioritized using a RICE (Reach, Impact, Confidence, Effort)
  1873→  1873→framework, and incorporated into the next sprint. A centralized feedback
  1874→  1874→tracker spreadsheet maintains a public changelog visible to all beta
  1875→  1875→participants, ensuring transparency and demonstrating responsiveness.
