  1000→prioritizes search speed and zero API latency while maintaining a clear
  1001→path for data updates. This approach recognizes that the Bangladesh
  1002→pharmaceutical market, while containing approximately 8,000 registered
  1003→brand-name products across 300+ generic molecules, is small enough to be
  1004→indexed entirely in client memory, yet large enough that manual curation
  1005→is impractical.
  1006→
  1007→[]{#_Toc100029 .anchor}**4.1 Data Source**
  1008→
  1009→The initial medicine database is sourced from the Directorate General of
  1010→Drug Administration (DGDA) of Bangladesh, which publishes a regularly
  1011→updated registry of all approved pharmaceutical products. This registry
  1012→includes brand names, generic compositions, dosage forms, strengths,
  1013→manufacturers, and retail prices. The raw DGDA data is processed through
  1014→an ETL (Extract, Transform, Load) pipeline that normalizes brand name
  1015→spellings, maps each brand to its generic molecule, and structures the
  1016→data into the JSON format required by Fuse.js. The pipeline is
  1017→version-controlled in the repository and can be re-run whenever the DGDA
  1018→publishes updated data.
  1019→
  1020→[]{#_Toc100030 .anchor}**4.2 Static JSON Dataset Architecture**
  1021→
  1022→Medicine data is stored as optimized static JSON files in the
  1023→/public/data/ directory. The primary file, medicines.json, contains an
  1024→array of objects, each with the following structure: { id, brand,
  1025→generic, dosageForm, strength, manufacturer, category, tags }. A
  1026→secondary file, generics.json, provides a mapping of generic molecule
  1027→names to their pharmacological categories, common side effects, and
  1028→typical dosage ranges. Both files are loaded once when the prescription
  1029→composer page mounts, and Fuse.js creates in-memory indices over the
  1030→brand, generic, and tags fields.
  1031→
  1032→The estimated file size for medicines.json is approximately 2MB for
  1033→8,000 entries, which compresses to under 400KB with Vercel\'s automatic
  1034→Brotli compression. This size is well within acceptable limits for a
  1035→one-time page-load fetch and has negligible impact on the user\'s
  1036→initial bandwidth, even on slow mobile connections common in Bangladesh.
  1037→
  1038→[]{#_Toc100031 .anchor}**4.3 Search Behavior**
  1039→
  1040→Fuse.js is configured with a threshold of 0.3 (balancing precision and
  1041→recall), keys indexed on brand (weight 2), generic (weight 1.5), and
  1042→tags (weight 1). This configuration ensures that a partial brand name
  1043→like \'amo\' returns \'Amoxicillin\' brands as top results, while a
  1044→generic search for \'paracetamol\' returns all paracetamol-containing
  1045→brands. Results are displayed in a dropdown with the brand name
  1046→highlighted and the generic composition shown as a secondary line,
  1047→enabling doctors to quickly identify the correct medication even when
  1048→multiple brands share similar names.
  1049→
  1050→[]{#_Toc100032 .anchor}**4.4 Data Update Pipeline**
  1051→
  1052→Updates to the medicine database follow a two-tier strategy. Tier 1 (Hot
  1053→Updates) uses a Supabase table called medicine_hotfixes that stores
  1054→corrections and additions submitted through the crowdsourced feedback
  1055→system. When the prescription composer loads, it fetches any hotfixes
  1056→created since the last JSON deployment date and merges them into the
  1057→in-memory Fuse.js index. This ensures that critical corrections (e.g., a
  1058→dangerous dosage error) are available within minutes without requiring a
  1059→full deployment. Tier 2 (Cold Updates) involves periodic re-runs of the
  1060→DGDA ETL pipeline, regeneration of the static JSON files, and deployment
  1061→of a new version via Vercel. This happens on a monthly cadence or when
  1062→the hotfix table accumulates more than 50 entries, at which point the
  1063→hotfixes are folded into the main JSON dataset.
  1064→
  1065→[]{#_Toc100033 .anchor}**4.5 Crowdsourced Feedback Workflow**
  1066→
  1067→The prescription composer includes a feedback button that allows doctors
  1068→to flag incorrect or missing medicine data. When clicked, it opens a
  1069→lightweight form pre-populated with the current medicine name and a
  1070→dropdown for feedback type (missing, incorrect, duplicate, suggestion).
  1071→The submission is stored in the medicine_feedback table with a status of
  1072→\'pending\'. An admin dashboard, accessible only to the RxBD operations
  1073→team, displays pending feedback items sorted by submission date.
  1074→Reviewed items are either resolved (by creating a hotfix entry in the
  1075→medicine_hotfixes table) or rejected (with a reason recorded in the
  1076→audit_log). The submitting doctor receives a notification when their
  1077→feedback is resolved, encouraging continued participation in the
  1078→improvement process.

  1124→[]{#_Toc100036 .anchor}**5.2 Monthly Prescription Counter Mechanism**
  1125→
  1126→The 50-prescription monthly limit for Free-tier users is enforced
  1127→through a combination of database-level and application-level checks.
  1128→The subscriptions table maintains a prescriptions_this_month counter and
  1129→a counter_reset_at date. Every time a Free-tier doctor creates a new
  1130→prescription (transitions from draft to final), the API route increments
  1131→the counter. Before allowing the creation, it checks whether the counter
  1132→has reached the 50-prescription limit. If counter_reset_at is older than
  1133→30 days, the counter is reset to 1 and the date is updated to the
  1134→current date.
  1135→
  1136→A scheduled Supabase Edge Function runs daily at midnight Bangladesh
  1137→time (UTC+6) to reset all Free-tier counters where counter_reset_at is
  1138→older than 30 days. This ensures that the counter resets even if a
  1139→doctor does not visit the platform on their reset date. Additionally,
  1140→the RLS policy on the prescriptions table includes a subquery check
  1141→against the subscriptions table, providing a secondary enforcement layer
  1142→that prevents counter bypass through direct database access.
  1143→
  1144→[]{#_Toc100037 .anchor}**5.3 Free-to-Premium Upgrade Flow**
  1145→
  1146→When a Free-tier doctor approaches their monthly limit (at 40
  1147→prescriptions), a non-intrusive banner appears at the top of the
  1148→prescription composer reading \'You have 10 prescriptions remaining this
  1149→month. Upgrade to Premium for unlimited access.\' At 48 prescriptions,
  1150→the banner becomes more prominent with a yellow highlight. At 50
  1151→prescriptions, the composer is disabled and replaced with a full-screen
  1152→upgrade prompt that explains the Premium benefits and includes a direct
  1153→\'Upgrade Now\' button. This graduated approach provides multiple
  1154→touchpoints for conversion without alienating users with hard paywalls
  1155→at the first sign of friction.
  1156→
  1157→The upgrade flow itself redirects the doctor to a Stripe Checkout-like
  1158→page powered by SSLCommerz. After successful payment, a webhook from
  1159→SSLCommerz hits the /api/webhooks/sslcommerz API route, which verifies
  1160→the payment signature using the store\'s secret key, updates the
  1161→subscriptions table to set plan=\'premium\', records the SSLCommerz
  1162→transaction ID, and sets the premium_expires_at date. The doctor\'s JWT
  1163→is refreshed on their next request to include the updated plan claim,
  1164→immediately unlocking Premium features.
  1165→
  1166→[]{#_Toc100038 .anchor}**5.4 Payment Integration via SSLCommerz**
  1167→
  1168→SSLCommerz is the leading payment gateway in Bangladesh, supporting all
  1169→local payment methods including bKash, Nagad, Rocket, DBBL Nexus, and
  1170→international Visa/Mastercard. Integration follows SSLCommerz\'s hosted
  1171→checkout flow: the RxBD backend generates a payment session by calling
  1172→SSLCommerz\'s /gwprocess/v4/api.php endpoint with the doctor\'s
  1173→information and payment amount, receives a redirect URL, and sends the
  1174→doctor to SSLCommerz\'s hosted payment page. After payment completion,
  1175→SSLCommerz sends an IPN (Instant Payment Notification) webhook to the
  1176→/api/webhooks/sslcommerz endpoint, which is the authoritative source of
  1177→payment confirmation. The Premium tier is priced at BDT 499 per month
  1178→(approximately USD 4.50), a price point validated through interviews
  1179→with 15 pilot doctors who indicated willingness to pay between BDT 300
  1180→and BDT 800 for premium prescription features.
  1181→
  1182→[]{#_Toc100039 .anchor}**5.5 Downgrade and Cancellation Policy**
  1183→
  1184→When a Premium subscription expires (premium_expires_at passes without
  1185→renewal), the system automatically reverts the doctor to the Free tier.
  1186→The prescriptions_this_month counter is initialized to the count of
  1187→prescriptions created in the current calendar month, ensuring that the
  1188→doctor does not receive a fresh 50-prescription allocation upon
  1189→downgrade. Existing Premium features (custom letterhead, digital
  1190→signature) are preserved in the doctor_settings table but are no longer
  1191→rendered in the PDF output. The doctor\'s data, including all
  1192→prescription history, remains fully accessible. A 3-day grace period
  1193→after expiration allows continued Premium access while a renewal
  1194→reminder is sent via email and in-app notification.

  1439→[]{#_Toc100051 .anchor}**7.4 Onboarding & Trust**
  1440→
  1441→The onboarding experience is designed to achieve the \'Aha!\' moment
  1442→within 5 minutes of signup. A 3-step guided walkthrough leads the new
  1443→doctor through: Step 1 (Complete Your Profile) entering their name,
  1444→degrees, specialty, and chamber information; Step 2 (Customize Your
  1445→Prescription) uploading a logo and configuring the header layout with a
  1446→live preview; Step 3 (Write Your First Prescription) completing a test
  1447→prescription with a dummy patient to experience the full workflow. Each
  1448→step includes contextual tooltips and a progress indicator. The
  1449→walkthrough can be skipped and resumed later from the Settings page, but

  1454→anti-forgery safeguard. Every prescription PDF includes a unique QR code

  1456→rxbd.com.bd/verify/\[prescription_id\]. Scanning the QR code opens a
  1457→public, read-only verification page that displays the doctor\'s name,
  1458→patient\'s name, visit date, and a tamper-evident status indicator
  1459→(green checkmark for verified, red warning if the prescription has been
  1460→modified since issuance). The verification page is implemented as a
  1461→server-side rendered Next.js page with ISR (Incremental Static
  1462→Regeneration) for performance, using a 60-second revalidation period to
  1463→balance freshness with cache efficiency.

  1465→[]{#_Toc100052 .anchor}**7.5 Internationalization (i18n)**
  1466→
  1467→RxBD supports two languages: English (default) and Bengali (Bangla). The
  1468→interface language is determined first by the user\'s browser
  1469→Accept-Language header, then by the locale preference stored in the
  1470→doctor\'s profile, and finally by a manual toggle in the navigation bar.
  1471→All UI strings, form labels, button text, error messages, and onboarding
  1472→content are externalized into message catalogs using next-intl. The
  1473→prescription content itself (chief complaints, diagnosis, medication
  1474→instructions) is entered by the doctor in their preferred language and
  1475→is not auto-translated, as medical terminology requires precision that
  1476→machine translation cannot guarantee. The PDF template layout adapts to
  1477→the selected locale, adjusting for Bengali text direction
  1478→(left-to-right, same as English) and font rendering using Noto Sans
