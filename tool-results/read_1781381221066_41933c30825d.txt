    36→   535→[]{#_Toc100011 .anchor}**2.6 Data Resilience**
    37→   536→
    38→   537→Data resilience in RxBD operates at three levels, ensuring that no
    39→   538→single point of failure can result in permanent data loss.
    40→   539→
    41→   540→[]{#_Toc100012 .anchor}**2.6.1 Local-First Auto-Save**
    42→   541→
    43→   542→Prescription drafts are automatically persisted to the browser\'s
    44→   543→localStorage every 10 seconds using a debounced Zustand middleware. The
    45→   544→auto-save captures the full prescription state including patient
    46→   545→context, medications added, clinical notes, and form dirty state. Upon
    47→   546→reconnection after a connectivity loss, the application detects local
    48→   547→drafts by comparing the localStorage timestamp against the server\'s
    49→   548→last-known save time and prompts the user to resume their work. This
    50→   549→mechanism prevents data loss even during extended internet outages,
    51→   550→which are common in semi-urban Bangladesh, and requires zero server
    52→   551→infrastructure.
    53→   552→
    54→   553→[]{#_Toc100013 .anchor}**2.6.2 Database Backup Strategy**
    55→   554→
    56→   555→The primary backup mechanism uses Supabase\'s built-in daily automated
    57→   556→backups, which are available on the Pro plan. During the free-tier
    58→   557→launch phase, a GitHub Actions workflow performs a daily pg_dump of the
    59→   558→Supabase database, encrypts the dump using AES-256 encryption with a key
    60→   559→stored in GitHub Secrets, and commits the encrypted backup to a private
    61→   560→repository. This approach provides 7-day retention without additional
    62→   561→cost. The encryption key is managed separately from the backup storage,
    63→   562→ensuring that even if the repository is compromised, the backup data
    64→   563→remains unreadable without the key. On migration to Supabase Pro, the
    65→   564→daily automated backups with 7-day Point-In-Time Recovery become the
    66→   565→primary mechanism, and the GitHub Actions backup is retained as a
    67→   566→secondary off-site copy.
    68→   567→
    69→   568→[]{#_Toc100014 .anchor}**2.6.3 Subscription Gating**
    70→   569→
    71→   570→Access control and tier-based feature restrictions are enforced via
    72→   571→Next.js Middleware, which runs on Vercel\'s edge network before any page
    73→   572→renders. The middleware extracts the Supabase Auth JWT from the request
    74→   573→cookie, decodes the user\'s plan metadata (stored as a custom claim in
    75→   574→the JWT), and checks the current prescription count against the plan
    76→   575→limit. If a Free-tier user has reached their 50-prescription monthly
    77→   576→limit, the middleware redirects them to an upgrade prompt page rather
    78→   577→than the prescription composer. Premium features such as custom
    79→   578→letterhead branding and digital signature upload are gated by checking
    80→   579→the plan claim before rendering the relevant UI components, and API
    81→   580→routes perform a secondary server-side plan check to prevent client-side
    82→   581→bypass.
    83→   582→
    84→   583→[]{#_Toc100015 .anchor}**3. Database Schema**
    85→   584→
    86→   585→The database is structured to support multi-tenancy where each doctor
    87→   586→owns their specific settings and patient records. The schema uses
    88→   587→Supabase PostgreSQL with Row Level Security (RLS) enforced on every
    89→   588→table. All tables include standard audit columns (created_at,
    90→   589→updated_at) and use UUID primary keys for global uniqueness and to
    91→   590→prevent enumeration attacks.
    92→   591→
    93→   592→[]{#_Toc100016 .anchor}**3.1 Entity Relationship Overview**
    94→   593→
    95→   594→The core data model revolves around the doctor entity as the tenant
    96→   595→root. Each doctor owns their settings, patients, and prescriptions. A
    97→   596→patient belongs to exactly one doctor (to maintain data isolation), and
    98→   597→a prescription belongs to one patient and one doctor. The subscriptions
    99→   598→table tracks the doctor\'s current plan and usage counters, while the
   100→   599→medicine_feedback table supports crowdsourced improvement of the
   101→   600→medicine database.

   452→   951→[]{#_Toc100025 .anchor}**3.3 Row Level Security Policies**
   453→   952→
   454→   953→Every table in the database is protected by RLS policies ensuring that a
   455→   954→logged-in doctor can only select, insert, or update data where the
   456→   955→doctor_id matches their authenticated session UID. The RLS policies are
   457→   956→defined as follows for each table, using the auth.uid() function to
   458→   957→extract the current user\'s ID from the JWT token. For the doctors
   459→   958→table, the policy verifies that the id column matches auth.uid(). For
   460→   959→all other tables with a doctor_id column, the policy checks that
   461→   960→doctor_id equals auth.uid(). The audit_log table is write-only for
   462→   961→doctors (INSERT allowed, SELECT restricted to admin roles) to prevent
   463→   962→tampering with activity records.
   464→   963→
   465→   964→Additionally, RLS policies enforce subscription limits on the
   466→   965→prescriptions table. An INSERT policy on prescriptions checks the
   467→   966→subscriptions table to verify that the doctor has not exceeded their
   468→   967→monthly prescription limit before allowing a new prescription to be
   469→   968→created. This database-level enforcement prevents bypass through API
   470→   969→route vulnerabilities or direct Supabase client access.
   471→   970→
   472→   971→[]{#_Toc100026 .anchor}**3.4 Indexes**
   473→   972→
   474→   973→The following indexes are created to optimize the most frequent query
   475→   974→patterns. An index on patients(doctor_id, name) supports the patient
   476→   975→search dropdown in the prescription composer. An index on
   477→   976→prescriptions(doctor_id, created_at DESC) enables efficient history
   478→   977→retrieval sorted by most recent visit. An index on
   479→   978→prescriptions(patient_id, created_at DESC) supports the patient history
   480→   979→view. An index on subscriptions(doctor_id) provides fast lookups for
   481→   980→subscription gating middleware. A GIN index on
   482→   981→prescriptions(medications) allows JSONB queries for analytics (e.g.,
   483→   982→most prescribed medications). An index on medicine_feedback(status)
   484→   983→supports the admin review queue for pending feedback items.
   485→   984→
   486→   985→[]{#_Toc100027 .anchor}**3.5 Database Migrations**
   487→   986→
